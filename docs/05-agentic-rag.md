# 第 5 章：Agentic RAG

第五章开始把前几章真正串起来了。

如果说第四章讲的是“agent 会用工具”，那第五章讲的是：

**agent 如何把“检索外部知识”这件事，做成一个可迭代、会自我修正的过程。**

你可以先记一句总纲：

**普通 RAG 是“先查再答”，Agentic RAG 是“边想边查，查得不够就继续查，直到足够好再答”。**

## 1. 什么是 RAG，什么又是 Agentic RAG

### 普通 RAG

通常是固定流水线：

1. 用户提问
2. 系统检索相关文档
3. 把文档片段塞给模型
4. 模型生成答案

### Agentic RAG

让 agent 自己决定：

- 要不要检索
- 先检索什么
- 检索结果够不够
- 要不要改写查询
- 要不要换一个数据源
- 要不要再查一次
- 什么时候可以收尾回答

所以第五章最核心的一句话是：

**Agentic RAG 的“agentic”，不在于用了检索，而在于它对检索过程本身拥有决策权。**

## 2. 为什么普通 RAG 不够

普通 RAG 的问题主要有几个：

- 第一次检索不准，就整体偏掉
- 查询写得不好，召回结果就差
- 用户问题复杂时，一次检索往往不够
- 多数据源场景下，固定流程不灵活
- 如果结果不完整，系统通常不会主动补查

所以 Agentic RAG 的出现，本质上是为了解决：

**“一次检索不够用”的问题。**

## 3. Agentic RAG 的定义

Agentic RAG 是一种 AI 范式，LLM 不只是从外部数据源取信息，还会**自主规划下一步**。它通过：

- LLM 调用
- tool/function 调用
- structured output
- 结果评估
- 查询重写
- 多轮迭代

不断循环，直到结果满意。

## 4. 什么叫 maker-checker

可以把它理解成两步循环：

- `maker`
  先做一个初步答案或初步检索结果

- `checker`
  再验证它、补全它、纠正它

本质是：

**先产出候选，再二次验证。**

## 5. owning the reasoning process

第五章里一句很重要的话是：

**真正让系统变得 agentic 的，是它“拥有自己的推理过程”。**

不是人把检索步骤完全写死，而是模型自己根据当前结果决定：

- 结果够不够？
- 查询要不要改写？
- 要不要换数据源？
- 还需不需要进一步确认？

## 6. Agentic RAG 的标准循环

1. 用户给目标
2. 模型判断缺什么信息
3. 调用检索工具
4. 评估结果够不够
5. 不够就改查询或换工具继续查
6. 记住之前试过什么
7. 满意后再生成最终答案

所以它更像：

**LLM -> retrieval/tool -> assessment -> refinement -> repeat -> final answer**

## 7. memory 和 state 为什么重要

没有状态，agent 就可能：

- 一直重复同一查询
- 一直走错方向
- 反复检索相同信息
- 无法从失败中修正

所以 Agentic RAG 能不能好用，很大程度上取决于：

**它是不是记得之前已经试过哪些路。**

## 8. failure mode 和 self-correction

系统遇到问题时可以：

- 重新查询
- 改写查询
- 看其他数据集
- 调诊断工具
- 必要时请求人工介入

所以 Agentic RAG 的价值就在于：

**不是第一下失败就直接输出糟糕答案，而是尝试补救。**

## 9. notebook 里是怎么落地的

Python notebook 用了一个很轻量的旅行知识库例子，主要做了三件事：

1. 把知识源包装成工具
2. 给 agent 一个“先查再答”的规则
3. 引入 maker-checker agent

这说明：

**在 Agentic RAG 里，检索本身被看作一种 tool use。**

## 10. Agentic RAG 和普通搜索增强问答的本质区别

- 普通 RAG：检索是固定前置步骤
- Agentic RAG：检索是 agent 在推理过程中动态决定的动作

## 11. 这一章适合哪些场景

- 法规分析
- 企业知识问答
- 复杂报表分析
- 研究助手
- 医疗/金融类证据检索辅助

不太适合：

- 问题很简单
- 一次检索足够
- 成本和延迟特别敏感

## 12. 你学完第五章，应该真正得到什么

- 检索不一定是固定流程，也可以是 agent 的自主动作
- 好的知识增强系统不只是“查”，还要“评估查得够不够”
- 多轮检索和自我修正，是 Agentic RAG 的核心
- Agentic RAG 本质是把第四章的 tool use 用在“知识获取”上

## 13. 学习版总结

- 普通 RAG 是“先检索再生成”
- Agentic RAG 是“检索、评估、重试、再检索、再生成”
- 它的关键是 agent 对检索过程本身有决策权
- maker-checker 是最核心的工作方式
- 它更适合复杂、正确性优先、多步研究型任务
