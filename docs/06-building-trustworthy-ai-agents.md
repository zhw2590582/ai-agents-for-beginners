# 第 6 章：Building Trustworthy AI Agents

第六章是前面几章的“刹车系统”。

前面你学到的是：

- agent 是什么
- framework 是什么
- tool use 怎么做
- Agentic RAG 怎么迭代检索

第六章开始强调：

**agent 不只是要能做事，还必须在做事时可控、安全、守边界。**

先记一句总纲：

**一个强大的 agent，如果不可信，通常就不能真正上线。**

## 1. 第六章在讲什么

这一章核心不是“再加一个新模式”，而是在讲 agent 系统的三个底线：

- `Safety`
- `Security`
- `Privacy`

也就是：

- 它会不会做错事
- 它会不会被攻击
- 它会不会泄露不该泄露的数据

## 2. 什么叫 trustworthy

trustworthy 在这一章里不是抽象地说“让用户喜欢它”，而是很具体：

- 按预期工作
- 不轻易被 prompt 注入带偏
- 不乱碰敏感系统
- 不制造高额成本或资源滥用
- 不基于被污染的数据胡乱输出
- 出错时不会连锁扩散
- 在高风险场景下有人类兜底

## 3. system message 被提升到了安全层

这一章强调：

**system message 不只是行为说明，也是安全边界的一部分。**

它提出了一个 `system message framework` 的思路：

1. 先写一个 `meta system message`
2. 再写一个基本业务描述
3. 让模型帮你生成更完整、更结构化的系统消息
4. 反复迭代优化

## 4. 为什么 system message 在 agent 里更重要

因为普通聊天模型就算答偏了，很多时候只是“说错话”。

但 agent 可能会：

- 调工具
- 查内部系统
- 改数据
- 执行多步流程

这时如果 system message 很弱，模型可能会：

- 被用户诱导越权
- 跳过原本应该确认的步骤
- 忘记拒绝高风险操作
- 在证据不足时强行作答

## 5. 这一章列出的几类核心威胁

### Task and Instruction

攻击者试图改写 agent 的目标或指令。

### Access to Critical Systems

攻击者试图借 agent 间接拿到不该拿的数据或能力。

### Resource and Service Overloading

通过 agent 高频调用工具，消耗成本或打崩服务。

### Knowledge Base Poisoning

不是直接攻击 agent，而是污染 agent 使用的数据源。

### Cascading Errors

一个错误扩散成多个系统问题。

## 6. Human-in-the-loop 为什么在这章出现

一旦系统有以下特征：

- 会调用工具
- 会执行动作
- 会访问敏感数据
- 会长链路运行

就不能指望“模型自己永远判断正确”。

所以 human-in-the-loop 的作用就是：

**在关键节点保留人工审批权。**

常见适用场景：

- 支付和扣费
- 发邮件和对外沟通
- 删除/修改数据
- 法务、医疗、金融等高风险建议
- 连续失败后的升级处理

## 7. 这一章其实在回答一个关键问题

既然 agent 的能力都来自外部系统接入，那可信性的核心，就不只是模型，而是整套：

**模型 + 工具 + 数据 + 权限 + 审批**

的系统设计。

## 8. 应主动意识到的可信设计清单

- 系统消息是否清楚定义了边界？
- 是否存在 prompt injection 防护？
- 工具权限是不是最小化？
- 敏感操作是否需要审批？
- 数据源是否可信且可审计？
- 是否有限流、限额、重试和超时控制？
- 是否有日志、追踪和回放能力？
- 是否在高风险场景保留人工复核？

## 9. notebook 示例真正想教你的是什么

重点不是技术炫技，而是方法论：

- 不要随手写一段 system prompt 就结束
- 把 system prompt 设计流程标准化
- 用 meta-prompt 帮助你生成更结构化的 agent 规则
- 把角色、职责、行为方式、语气、限制写清楚

## 10. 你学完第六章，应该真正得到什么

- agent 的风险面比普通聊天模型大得多
- 风险来自提示词、工具、权限、数据源、外部系统和执行链路
- system message 是安全边界的一部分
- human-in-the-loop 是高风险操作的重要兜底

## 11. 学习版总结

- agent 越能行动，就越需要 guardrails
- system prompt 不只是角色设定，也是安全控制面
- 风险主要来自指令劫持、权限滥用、资源滥用、知识污染、连锁错误
- 高风险动作要有人类在环
- trustworthy 的重点不是“绝对正确”，而是“有边界、可审计、可干预”
