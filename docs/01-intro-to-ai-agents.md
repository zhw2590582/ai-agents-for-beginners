# 第 1 章：Introduction to AI Agents

第一章的核心，不是在教你“怎么马上写一个很复杂的 agent”，而是在帮你先建立一个正确的判断框架：什么才算 AI Agent，它适合解决什么问题，以及一个最小可用 agent 由哪些部分组成。

你可以把这一章理解成一句话：

**AI Agent = 以 LLM 为大脑 + 能感知环境 + 能调用工具 + 能执行动作 + 有记忆/知识支撑的系统。**

## 这一章最重要的三个结论

1. 普通 LLM 主要负责“生成内容”，AI Agent 负责“完成任务”。
2. AI Agent 不是单个模型，而是一个系统。
3. 不是所有场景都该用 agent，只有在任务开放、多步、需要持续改进时，agent 才有优势。

## 1. 什么是 AI Agent

仓库里的定义很清楚：AI Agent 是一种让大语言模型能够执行动作的系统，它通过工具和知识扩展 LLM 的能力。

这里面有 5 个关键词：

- `System`：agent 不是一句 prompt，而是一套协同工作的部件。
- `LLM`：模型负责理解语言、推理和规划。
- `Actions`：agent 不只会“说”，还会“做”。
- `Tools`：agent 借助函数、API、数据库、搜索等能力行动。
- `Memory + Knowledge`：agent 不只依赖当前对话，还能结合历史偏好或外部知识。

这一章里用“旅行代理”举例非常好理解：

- 环境：旅行预订系统
- 传感器：酒店空房、机票价格、用户偏好
- 执行器：下单、取消、查询、推荐
- LLM：理解“我想去一个温暖的海边城市”
- 工具：查目的地、查价格、预订服务

所以你要记住一个判断标准：

**如果一个系统只是回答问题，它更像聊天模型；如果它能根据目标调用工具并改变外部世界，它才更像 agent。**

## 2. AI Agent 和普通聊天机器人有什么区别

这是初学者最容易混淆的点。

普通聊天机器人通常是：

- 你问一句，它答一句
- 重点是文本生成
- 不一定能访问外部系统
- 不一定有任务闭环

AI Agent 通常是：

- 接收目标，不只是接收问题
- 会拆步骤
- 会调用工具
- 会根据环境反馈继续调整
- 最终追求“任务完成”

举个简单对比：

- 聊天机器人：告诉你“东京适合旅行，因为……”
- AI Agent：查询候选目的地，结合你的偏好筛选，甚至进一步生成行程建议

## 3. 第一章介绍了哪些 Agent 类型

这一部分你先不用死记定义，先理解“复杂度升级路线”就行：

- `Simple Reflex Agent`
  按固定规则立刻反应。像“检测到投诉邮件就转给客服”。

- `Model-Based Reflex Agent`
  不只看当前输入，还考虑环境状态。像“参考历史价格变化决定优先推荐哪条线路”。

- `Goal-Based Agent`
  围绕目标规划步骤。像“帮我完成从当前城市到目的地的完整出行安排”。

- `Utility-Based Agent`
  会权衡得失。像“在便宜和方便之间找最优解”。

- `Learning Agent`
  会根据反馈变好。像“根据用户旅行后的满意度逐步优化推荐”。

- `Hierarchical Agent`
  上层 agent 负责任务拆分，下层 agent 执行子任务。

- `Multi-Agent System`
  多个 agent 协作或竞争完成任务。

你可以把它们理解为一条演进线：

**规则反应 -> 结合状态 -> 围绕目标规划 -> 做权衡优化 -> 能学习 -> 多 agent 协作**

## 4. 什么情况下该用 AI Agent

这一章给了 3 个典型场景，这个非常重要，因为它决定你是不是在“过度设计”。

适合用 agent 的场景：

- `Open-Ended Problems`
  任务步骤不固定，无法完全硬编码。
- `Multi-Step Processes`
  需要多轮决策、调用多个工具、逐步推进。
- `Improvement Over Time`
  系统需要根据反馈不断优化。

反过来说，不太适合的场景通常是：

- 规则非常固定
- 一次查询就能完成
- 没有工具调用需求
- 不需要规划和反馈闭环

一个实用判断法：

**如果 `if/else + API` 就能稳定解决，大概率先别上 agent。**
**如果任务目标明确，但路径不固定，而且需要工具和反馈，那 agent 才有价值。**

## 5. 第一章讲的 Agentic Solution 基础搭建块

这一章最后其实在给后面课程铺路，讲了 3 个基础概念：

- `Agent Development`
  先定义 agent 有哪些工具、动作、行为。

- `Agentic Patterns`
  因为 agent 往往是多步任务，所以不能只靠一次 prompt，需要用模式来组织多轮决策。

- `Agentic Frameworks`
  框架帮助你把这些模式写成工程化系统，并提升可观测性、调试能力、扩展性。

## 6. notebook 示例在做什么

第一章的 Python 示例其实非常“最小化”，但很适合入门。

它做了 4 件事：

1. 连接 Azure AI Foundry Agent Service

这说明一个最小 agent 的结构其实很清楚：

- 一段角色说明
- 一个或多个工具
- 一个运行入口
- 一条用户任务

## 7. 你学完这一章，应该真正获得什么

不是“我会背 AI Agent 定义了”，而是以下 3 个能力：

- 你能分辨一个系统是不是 agent
- 你能判断一个场景适不适合 agent
- 你知道一个最小 agent 至少需要：`LLM + instructions + tools + environment context`

## 8. 学习版总结

- AI Agent 不等于聊天机器人
- Agent 的核心是“为了目标采取行动”
- 工具是 agent 能力边界的关键
- 记忆和知识让 agent 更像一个持续工作的系统
- 多步、开放式、有反馈的任务最适合 agent
- 框架的作用是把 agent 从 demo 变成工程
